# 任务：生成 `continuity_log.md`

你将收到：

- 用户需求（可能不完整）
- `continuity_log.md` 模板（需要严格遵循结构与表格列）
- `bible.md` / `characters.md` / `outline.md`（用于对齐禁改事实、线索、人物状态）

要求：

- 用中文输出完整 `continuity_log.md` 内容，严格遵循模板结构。
- `A) 禁改事实表`：把 pre-story 与已确定设定写成可审计条目（F-xxx）。每新增一条事实，必须写清“可指认点”（比如：见证人/物证/账目/口供/伤痕/契约/现场）。
- `B) 未解线索表`：列出所有伏笔（CL-xx）并标注 planned（规划）/ seeded（已埋）/ open（已开启）/ resolved（已回收）/ burned（已暴露不可再用）；不要让线索无声消失。
    > 规则：
    > - 线索必须是“实体”：记录、证据、债务、物件、目击、承诺、伤痕、口供……不能只是“某人想法”。
    > - 每条线索必须挂靠至少一条剧情线（D-xx），否则视为无意义伏笔。
    > - 线索不能“无声消失”：若暂时搁置，状态必须仍为 seeded/open，并写清楚是谁压住、凭什么压得住、压不住会怎样。
- `C) 人物状态表`：列出每个主要人物的心理/关系/债务状态基线（后续每章要更新，但不允许漂移）。
    > 规则：
    > - 债务必须写“凭据”（谁知道、凭什么能追讨、追讨会用什么方式兑现）。
    > - 伤疤必须可指认（某个承诺、某段公开冲突、某个伤势、某个记录、某个名声污点）。
    > - “计时器/触发点”必须是可触发条件，而不是情绪描述。
- `D) 剧情线账本（短/中/长线必须可审计）`： 列出长中短线的剧情推进
    > 规则：
    > - 每章至少要推进 1 条短线（S）：让其进入 running 或 escalating，并更新“最近一次推进章”为本章。
    > - 每章的“新增不可逆后果”必须落到这里：新增一条 D 线，或让某条 D 线的“可利用物/压力/计时器”更具体、更难拆。
    > - 中线（M）不能一直 armed：必须在若干章后进入 converging，并在幕末出现 detonated（爆点发生）。爆点发生后，要么 closed，要么 transformed（并转挂到更大线）。
    > - “下一次必须回应的点”必须能被验证为“有人能拿着它做事”（索取、威胁、交换、阻断、嫁祸、逼选）。
- `E) 章节更新记录`： 将outline与continuity_log对应

输出：只输出 `continuity_log.md`（不要代码块）。
