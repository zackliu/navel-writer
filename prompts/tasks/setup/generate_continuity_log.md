# 任务：生成 `continuity_log.md`

你将收到：
- 用户需求（可能不完整）
- `continuity_log.md` 模板（需要严格遵循结构与表格列）
- `bible.md` / `characters.md` / `outline.md`（用于对齐禁改事实、线索、人物状态、收口句、代价换形）

输出要求：
- 用中文输出完整 `continuity_log.md` 内容，严格遵循模板结构与表格列，不要代码块。
- 必须与 `outline.md` 完全对齐：章节编号、关键事件、每章“收口句”的含义（软/硬）、卷级爆点与“结构性锁死”、代价类型与代价换形，均要能在账本里被查询和验证。
- 目标是“可查询和验证的连续性系统”：任何一条压力/代价/门/线索都必须能回答：
  - 谁掌握了什么（凭据是什么）？
  - 它如何被用来做事（索取/威胁/交换/阻断/嫁祸/逼选）？
  - 何时/在什么条件下会触发或升级？

生成目标（明确写出来，必须执行）：
- 账本必须服务于 outline 的“收口递进”：
  - 每章至少记录一个“门”的变化：硬收口落到 A/B/ D 表里；软收口落到 C 表（人物策略/防御更新）并在 E 表可追溯。
- 账本必须服务于“代价类型 + 换形防崩溃”：
  - 在 D 表必须能看到主代价如何变贵、何时换形、换形前后的行动风格如何改变（用“可利用物/压力/计时器/下一次必须回应”来体现）。

================================
# 强制对齐规则（必须写进表格内容，不允许“放在心里”）

1) 对齐 outline 的“收口句”
- 若某章为硬收口：必须在 A 表（禁改事实）或 B 表（线索实体）中出现对应条目（F-xx 或 CL-xx），并在 D 表的某条线（D-xx）的“可利用物/压力/计时器/NEXT”中体现其可用性；同时在 E 表把该章对应的 F/CL/D 写清楚。
- 若某章为软收口：必须在 C 表把该人物的“防御/策略更新”写成可验证句式（从此在X情境下会Y/不再Z），并在 D 表的某条线（D-xx）的“当前压力/NEXT”体现这种策略如何改变对抗形态；同时在 E 表标注该章引发的策略更新（写入“新增不可逆代价”一列即可）。

2) 对齐 outline 的“爆点=结构性锁死”
- 每卷爆点必须在 A/B/D 表同时有落点：
  - A：至少一条“已发生不可反悔”的事实（F-xx）或明确的“记录/处分/契约/公开表态/资源塌陷/计时器启动”；
  - B：若爆点涉及把柄/证据/目击/物件，必须是实体线索（CL-xx）且标注当前持有人；
  - D：至少一条中线 M（D-xx）状态变为 detonated，并说明它如何锁死后续选择空间。

3) 对齐 outline 的“代价类型 + 代价换形”
- 你必须在 D 表中明确标注（写进“可利用物/当前压力/计时器/NEXT”字段）：
  - 本书主要代价类型是什么（信息/安全/身份/关系/资源/时间/能力/因果等）；
  - 哪些线（D-xx）主要在支付哪类代价；
  - 当某类代价付到极限时，发生了哪次“换形”（A→B），以及换形后人物行动风格怎么变（可观察的策略变化要落在 C 表）。

# 表格填写规则（逐表）

A) 禁改事实表（发生过就不能反悔）
- 只记录“已经发生”的事实（含 pre-story 与各章新增）。
- 每条事实必须是可查询和验证条目（F-xxx），并必须写清“可指认点/凭据”：
  - 见证人、物证、账目、口供、伤痕、契约、监控、短信/聊天记录、纸质文件、系统记录等。
- 与 outline 的硬收口对齐：凡是章级硬收口，只要它“落地成事实”，就必须新增或更新一条 F-xxx。

B) 未解线索表（所有伏笔都在这里排队）
- 状态：planned（规划）/ seeded（已埋）/ open（已开启）/ resolved（已回收）/ burned（已暴露不可再用）
- 线索必须是“实体”：记录、证据、债务凭据、物件、目击、承诺文本、伤痕、口供……不能只是“某人想法/感觉”。
- 每条线索必须挂靠至少一条剧情线（D-xx），否则视为无意义伏笔。
- 线索不能“无声消失”：若暂时搁置，状态必须仍为 seeded/open，并写清：
  - 谁压住、凭什么压得住、压不住会怎样（用“当前持有人/知情者 + 备注”写清）。

C) 人物状态表（每章后更新但不能漂移）
- 这里必须承载“软收口”的更新：人物的防御/策略必须可验证。
- 债务必须写“凭据”（谁知道、凭什么能追讨、追讨会用什么方式兑现）。
- 伤疤/不可逆代价必须可指认（承诺、公开冲突、伤势、记录、名声污点、被谁掌握的把柄）。
- 计时器/触发点必须可触发（条件/窗口/截止），不能写情绪。

D) 剧情线账本（短/中/长线必须可查询和验证）
- 线类型：S（短线：章章在动的压力/计时） / M（中线：多章汇聚爆点） / L（长线：全书终局）
- 状态：armed / running / escalating / converging / detonated / closed / transformed
- 你必须让 D 表成为 outline 的“压力主干”：
  - 每章至少推进 1 条短线（S），让其进入 running 或 escalating，并更新“最近一次推进章”为本章。
  - 每章的“新增不可逆后果”必须落到这里：新增一条 D 线，或让某条 D 线的“可利用物/压力/计时器/NEXT”更具体、更难拆。
  - 中线（M）不能一直 armed：必须在若干章后进入 converging，并在卷末/幕末 detonated；爆点后要么 closed，要么 transformed（转挂到更大线并写清换形后的代价类型）。
  - “下一次必须回应的点”必须可验证为“有人能拿着它做事”（索取、威胁、交换、阻断、嫁祸、逼选）。

E) 章节更新记录（用于对照 outline 与账本）
- 必须逐章列出与 outline 对齐的信息：
  - 新增禁改事实（F-xx）
  - 新增/推进线索（CL-xx）
  - 新增不可逆代价（可写硬收口或软收口；若软收口要写成策略更新句式）
  - 推进的剧情线（D-xx）
  - 本章末尾悬置问题（必须对齐 D-xx 的 NEXT，且是可被人拿来做事的问题）

最终输出：
- 只输出 `continuity_log.md` 内容（不要任何额外说明）。
